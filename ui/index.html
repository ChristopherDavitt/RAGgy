01 — A local search engine for your own corpus

The things you’ve read, written,
and recorded. Searchable — and
never leaving your laptop.

RAGgy indexes your documents, notes, and code into a hybrid retrieval engine: exact keywords, meaning, and the entities that connect them. Ask a question in plain English. Get back the passages that matter — with the people, places, and dates in those passages already linked together.

02 — What you get

i.

Hybrid retrieval,
not just vectors.

BM25 for exact keyword matches, dense vector embeddings for meaning, and an entity graph for cross-reference. Merged into one ranked result. Better than grep, better than a vector DB alone.

ii.

Entities, linked
by the corpus itself.

People, organizations, dates, URLs, file paths, code identifiers — extracted on ingest, deduplicated, and connected through the passages they appear in. Ask who is connected to whom and get a real answer.

iii.

MCP-native.
Works with Claude.

Drop-in Model Context Protocol server. Claude Desktop, and any other MCP client, can search your corpus with zero additional glue. Also exposes an HTTP API and a plain CLI.

iv.

Private by design,
not by policy.

Everything — index, embeddings, processed documents — lives in ~/.raggy/. No cloud. No telemetry. No account. If your laptop is offline, RAGgy still works.

03 — Built for people who take their corpus seriously

RAGgy is a tool for working with your own material — not the open web, not a team’s shared drive, not someone else’s cloud. It's designed for the kinds of work where that distinction matters.

  1. Investigative journalists & researchers

    FOIA dumps. Leaked archives. Interview transcripts. Source material that cannot be uploaded to a commercial service without a real conversation about risk. RAGgy runs locally; the documents never leave the machine they were acquired on.

  2. Engineers with their own knowledge

    Codebases, design docs, architecture notes, past decisions. RAGgy indexes code with language awareness alongside prose, and finds the passage that answers your question even when your grep query was wrong.

  3. AI practitioners building agents

    Durable, queryable memory you control. Plug RAGgy in as an MCP server and your agent gets retrieval that actually understands the difference between keywords, meaning, and entities.

04 — How it works

Three indexes. One answer.

Full-text

BM25 · Tantivy

Rare terms, identifiers, exact phrases. The things dense vectors are famously bad at.

Semantic

HNSW · ONNX

Meaning-based retrieval via small local embeddings. Finds “login flow” when you searched for “authentication.”

Relational

Entity graph · SQLite

Every extracted person, place, date and identifier is a node in a local graph, linked through the passages they co-occur in.

At query time, all three signals are blended and filtered by the query’s metadata (type, date, scope). Documents connected to the entities mentioned in your query get a graph boost. Recent documents can be prioritised via exponential decay with a 30-day half-life. The full pipeline is documented in ARCHITECTURE.md.

05 — Install

Sixty seconds from
nothing to answers.

RAGgy is a single Rust binary. The first time you run it, it creates ~/.raggy/ and downloads the small ONNX models it needs (~200 MB). After that it’s entirely offline.

Requires Rust 1.75+. Binary releases and a Homebrew tap are on the roadmap. 

# Clone and install
git clone https://github.com/ChristopherDavitt/RAGgy
cd RAGgy
cargo install --path .

# First-time setup
raggy init

# Index a folder
raggy index ~/Documents/my-project

# Ask a question
raggy query "authentication middleware"
raggy query "pdfs about climate policy from last month"

06 — Claude Desktop, connected

// ~/Library/Application Support/Claude/claude_desktop_config.json
{
  "mcpServers": {
    "raggy": {
      "command": "raggy",
      "args": ["mcp"]
    }
  }
}

Drop it into
Claude Desktop.

RAGgy speaks the Model Context Protocol. Add the snippet on the left to Claude’s config, restart the app, and Claude can now search your indexed corpus as a native tool — without anything ever being sent to a third-party retrieval service.
